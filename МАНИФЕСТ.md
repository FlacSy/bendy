# Манифест проекта Bendy

*(Рабочее название)*

## 1. Суть проекта (Что мы строим)

**Bendy** — это архитектурный фреймворк и кодогенератор для Python, созданный для разработки бэкенда по канонам Domain-Driven Design (DDD).

Инструмент решает главную проблему современного бэкенда: выбор между *скоростью разработки* (как в Django/FastAPI-шаблонах) и *долгосрочной поддерживаемостью* (как в энтерпрайз-архитектурах). Мы даем скорость первого без ущерба для второго.

## 2. Идеология (Почему мы это делаем)

Современные "быстрые" фреймворки берут архитектурный кредит. Завязывая бизнес-логику на ORM и базу данных, они позволяют быстро запустить проект, но через год превращают его в монолитный клубок, который невозможно ни оптимизировать, ни распилить на микросервисы.

Bendy создается для того, чтобы автоматизировать рутину (создание слоев, мапперов, DTO), оставляя разработчику только самое важное — проектирование чистого домена и написание бизнес-логики.

## 3. Фундаментальные принципы (The Core Tenets)

* **Домен — это центр вселенной.** База данных, HTTP-роутеры и очереди сообщений — это лишь детали реализации (адаптеры). Слой бизнес-логики (`Domain`) ничего не знает про внешние технологии. Код домена — это чистый Python.
* **Явное разделение слоев (Onion Architecture).** Проект жестко делится на слои: Представление (Presentation) $\rightarrow$ Приложение (Application) $\rightarrow$ Домен (Domain) $\leftarrow$ Инфраструктура (Infrastructure). Нарушение направления зависимостей невозможно по дизайну.
* **Сгенерированное отдельно, ручное — отдельно.** Фреймворк генерирует рутинный код (файлы `.gen.py`: DTO, SQLAlchemy-модели, базовые репозитории). Этот код принадлежит машине и перезаписывается. Слой Use Cases и Domain-сущностей принадлежит разработчику и никогда не затирается.
* **Модульный монолит по умолчанию.** Каждый домен (`User`, `Order`, `Billing`) инкапсулирован в своей папке. Они общаются через строгие контракты (DTO). Если проект вырастает, любой модуль выносится в отдельный микросервис за пару часов без переписывания логики.
* **Никакой ORM-магии (Data Mapper).** Никакого ActiveRecord (как в Django). Инфраструктурный слой самостоятельно маппит таблицы SQLAlchemy в чистые доменные объекты Python. Проблема $N+1$ запросов контролируется явно.

## 4. Стратегия реализации (Как мы это делаем)

Проект состоит из двух связанных частей, которые разрабатываются итеративно:

### Фаза I: CLI-Scaffolder (Генератор)

* Отказ от ручного создания папок и boilerplate-кода.
* Утилита принимает команды (или манифесты на чистом Python) и разворачивает готовую структуру директорий.
* Генерация Pydantic-схем, заготовок под FastAPI-роутеры и пустых классов домена через Jinja2-шаблоны.

### Фаза II: Python-манифесты

* Создание DSL на базе Python.
* Разработчик описывает агрегат, его поля и типы данных в одном конфигурационном Python-файле.
* Генератор автоматически собирает миграции БД, Pydantic-контракты и классы маппинга (Domain $\leftrightarrow$ ORM).

### Фаза III: Runtime-Ядро (Фреймворк)

* Выделение общих DDD-механизмов в устанавливаемую библиотеку (`bendy.runtime`).
* Реализация умного паттерна **Unit of Work** (UoW) для транзакционной целостности без загрязнения бизнес-кода.
* Реализация **Domain Events** (событий домена) из коробки: сбор событий из доменных сущностей перед коммитом в БД и их автоматическая диспетчеризация (Event Bus).
